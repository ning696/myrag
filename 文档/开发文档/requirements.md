# RAG 对话机器人系统需求文档

**版本：** v1.0  
**编写日期：** 2026-06-02  
**项目名称：** myrag - 基于 RAG 的智能对话机器人  
**技术栈：** Spring Boot 3 + Vue 3 + TypeScript

---

## 1. 项目概述

### 1.1 项目目标

开发一个基于 RAG（Retrieval-Augmented Generation，检索增强生成）技术的智能对话机器人系统，支持：
- 用户注册和登录
- 上传文档并构建知识库
- 基于知识库内容的智能问答
- 可扩展的工具调用机制
- 有状态的多轮任务型对话（Skill 系统）

### 1.2 核心价值

- **准确性**：基于用户上传的文档知识库回答问题，减少 LLM 幻觉
- **可扩展性**：工具调用机制支持动态扩展功能（网络搜索、计算器、时间查询等）
- **任务型对话**：Skill 系统支持多轮对话完成复杂任务（如发送邮件、查询天气）
- **易用性**：简洁的 Web 界面，支持文档上传和实时对话

### 1.3 目标用户

- **企业内部知识管理**：HR、客服部门使用企业文档库回答常见问题
- **个人学习助手**：学生上传课程资料、论文，进行知识问答
- **开发者工具**：技术文档检索、代码库问答
- **研究人员**：学术论文检索和总结

### 1.4 项目范围

**第一版（MVP）包含：**
- 用户注册/登录系统（JWT 认证）
- RAG 核心功能（文档上传、向量化、智能问答）
- 工具调用机制（内置 `current_time`、`web_search`）
- Skill 系统（至少 2 个示例 Skill）
- 基础 Web 界面（登录注册、文档管理、对话界面）

**第一版不包含：**
- 多租户支持（不同组织隔离）
- 移动端应用
- 付费功能
- 第三方登录（微信、GitHub 等）

## 2. 功能需求

### 2.1 用户管理（优先级：P0）

#### 2.1.1 用户注册

**功能描述：**
新用户可以通过注册页面创建账号，注册后自动登录并跳转到主界面。

**用户故事：**
- 作为新用户，我希望能够注册账号，这样我可以拥有独立的知识库和对话历史
- 作为用户，我希望注册时密码强度有验证，保证账号安全

**验收标准：**
- ✅ 支持用户名、邮箱、密码注册
- ✅ 用户名唯一性验证（2-20 个字符，支持中英文、数字、下划线）
- ✅ 邮箱格式验证和唯一性验证
- ✅ 密码强度验证（至少 8 位，包含大小写字母和数字）
- ✅ 注册成功后自动登录并跳转到主界面
- ✅ 注册失败时显示明确的错误信息（用户名已存在、邮箱已注册等）

**技术约束：**
- 密码加密：BCrypt 算法
- 注册验证：使用 Spring Validation (`@Valid`)

#### 2.1.2 用户登录

**功能描述：**
已注册用户可以通过用户名/邮箱和密码登录系统。

**用户故事：**
- 作为注册用户，我希望能够登录系统，访问我的知识库和对话历史
- 作为用户，我希望登录状态能够保持 7 天，避免频繁登录

**验收标准：**
- ✅ 支持用户名或邮箱登录
- ✅ 密码错误提示（不显示用户名是否存在，防止用户枚举）
- ✅ 登录成功后返回 JWT Token
- ✅ 支持"记住我"功能（Token 有效期延长至 7 天）
- ✅ 连续登录失败 5 次后锁定账号 30 分钟

**技术约束：**
- JWT Token 有效期：默认 2 小时，勾选"记住我"后 7 天
- Token 存储：前端存储在 localStorage
- 密码验证：BCrypt 校验

#### 2.1.3 用户信息管理

**功能描述：**
用户可以查看和修改个人信息（昵称、头像、密码）。

**验收标准：**
- ✅ 查看当前用户信息（用户名、邮箱、注册时间）
- ✅ 修改昵称（2-20 个字符）
- ✅ 修改密码（需输入原密码验证）
- ✅ 上传头像（支持 JPG、PNG，最大 2MB）

#### 2.1.4 权限控制

**功能描述：**
系统支持两种角色：普通用户（USER）和管理员（ADMIN）。

**角色定义：**
- **USER**：普通用户，可以上传文档、进行对话
- **ADMIN**：管理员，可以配置工具、Skill，查看所有用户的文档

**权限矩阵：**
| 功能 | USER | ADMIN |
|------|------|-------|
| 注册/登录 | ✅ | ✅ |
| 上传文档 | ✅ | ✅ |
| 删除文档 | ✅（仅自己的） | ✅（所有） |
| 对话 | ✅ | ✅ |
| 配置工具 | ❌ | ✅ |
| 配置 Skill | ❌ | ✅ |
| 查看所有用户 | ❌ | ✅ |

**验收标准：**
- ✅ 未登录用户访问受保护接口时返回 401 错误
- ✅ 普通用户访问管理员接口时返回 403 错误
- ✅ 用户只能访问自己的文档和对话历史

### 2.2 RAG 核心功能（优先级：P0）

#### 2.2.1 文档上传与管理

**功能描述：**
用户可以上传文档到系统，系统自动处理并存储到知识库中。支持的文档格式包括 PDF、TXT、Markdown。

**用户故事：**
- 作为登录用户，我希望能够上传 PDF 文档，系统自动提取文本并向量化，这样我可以基于文档内容提问
- 作为登录用户，我希望能够查看已上传的文档列表，包括文件名、上传时间、处理状态
- 作为登录用户，我希望能够删除不再需要的文档，释放存储空间
- 作为用户，我希望只能看到自己上传的文档，不能访问其他用户的文档

**验收标准：**
- ✅ 支持上传 PDF、TXT、Markdown 格式文件
- ✅ 单个文件大小限制：50MB
- ✅ 上传后显示处理进度（处理中/已完成/失败）
- ✅ 文档列表展示文件名、类型、大小、上传时间、分块数量
- ✅ 支持删除文档（同时删除向量库中的数据）
- ✅ 上传失败时显示明确的错误信息（格式不支持、文件过大、解析失败等）

**技术约束：**
- 文档存储：文件系统或对象存储（第一版使用文件系统）
- 元数据存储：关系数据库（MySQL）
- 文档解析：Apache PDFBox（PDF）、Java 文本读取（TXT）、CommonMark（Markdown）
- 文档隔离：每个用户只能访问自己上传的文档

#### 2.2.2 文档处理与向量化

**功能描述：**
系统自动将上传的文档进行分块（chunk）、向量化（embedding），并存储到向量数据库中，为后续检索做准备。

**处理流程：**
```
文档上传 → 文本提取 → 文本分块 → 向量化（Embedding） → 存储到向量库
```

**技术要求：**
- **分块策略：**
  - Chunk 大小：500-1000 字符（中文）
  - 重叠部分：50-100 字符（保持上下文连贯性）
  - 分块单位：段落优先，避免在句子中间截断
- **向量化模型：**
  - 推荐：阿里云 DashScope `text-embedding-v2`（云端调用，中文优化，由 langchain4j-community-dashscope 集成）
  - 向量维度：1536（v1/v2）或 1024（v3）
- **元数据存储：**
  - 每个 chunk 包含：文档 ID、chunk 序号、原文内容、向量、来源位置
- **向量数据库：**
  - Milvus（企业级向量数据库，支持海量数据、高性能检索）

**验收标准：**
- ✅ 文档上传后自动触发分块和向量化流程
- ✅ 分块后的 chunks 数量合理（1000 字文档约 1-3 个 chunks）
- ✅ 向量化完成后更新文档状态为"已完成"
- ✅ 处理失败时更新状态为"失败"，并记录错误日志
- ✅ 支持查看文档的分块详情（chunk 数量、分块内容预览）

#### 2.2.3 智能问答

**功能描述：**
用户在对话界面输入问题，系统从知识库中检索相关内容，结合 LLM 生成准确的答案。

**用户故事：**
- 作为登录用户，我希望输入问题后能够获得基于我上传文档的准确回答，而不是通用的 LLM 回答
- 作为登录用户，我希望答案中包含引用来源（文档名称、位置），以便我验证答案的准确性
- 作为登录用户，我希望当知识库中没有相关信息时，系统明确告知我"未找到相关信息"
- 作为用户，我希望只能检索到自己上传的文档内容，不能访问其他用户的文档

**检索策略：**
1. 用户问题 → Embedding 模型 → 查询向量
2. 向量相似度检索 → 检索 Top-K chunks（K=3-5）
3. Chunks + 用户问题 → Prompt 模板 → LLM API → 生成答案
4. 返回答案 + 引用来源

**Prompt 模板：**
```
你是一个基于知识库的智能助手。请根据以下上下文回答用户问题。

上下文：
{context}

用户问题：{question}

回答要求：
1. 基于上下文回答，不要编造信息
2. 如果上下文中没有相关信息，明确告知用户"根据当前知识库，我无法找到相关信息"
3. 引用具体的文档来源（文档名称）
```

**验收标准：**
- ✅ 问题提交后 3 秒内返回答案（正常网络条件）
- ✅ 答案基于检索到的上下文，准确率 > 80%（通过测试用例验证）
- ✅ 答案包含引用来源（文档名称列表）
- ✅ 知识库无相关信息时，明确告知用户
- ✅ 支持多轮对话（保留会话上下文）
- ✅ 支持 Markdown 格式渲染（代码块、列表、加粗等）

**技术约束：**
- LLM API：DeepSeek（推荐）/ 智谱 GLM / 通义千问 / OpenAI
- 向量检索：Top-K=3（可配置）
- 超时设置：10 秒（LLM API 调用超时）
- 重试机制：最多 3 次

### 2.3 工具调用机制（优先级：P1）

#### 2.3.1 工具调用系统框架

**功能描述：**
工具是通过 LangChain4j `@Tool` 暴露给模型的后端能力。模型在 `TOOL_CALLING` 路由下自主决定是否调用 `current_time`、`web_search` 等工具，后端只负责按 `tools_config.enabled` 动态提供可用工具。

**工具调用生命周期：**
```
读取 tools_config 启停状态 → 组装已启用工具对象 → 构建 AI Service → 模型自动调用 @Tool → 汇总答案、引用和 usedTools
```

**配置管理：**
- 工具配置存储在数据库（`tools_config` 表）
- 支持运行时启用/禁用工具（无需重启服务）
- 工具运行参数不存数据库，统一由 `application.yml` 与环境变量维护

**验收标准：**
- ✅ 支持通过 Spring Bean 注册工具列表
- ✅ 支持动态启用/禁用工具（通过 API 接口）
- ✅ 工具执行失败不影响主流程（捕获异常并记录日志）
- ✅ 工具执行超时保护（由工具配置控制）
- ✅ 支持查询当前已加载的工具列表

#### 2.3.2 内置工具

##### current_time（当前时间工具）

**功能描述：**
返回当前日期、时间、星期和时区。系统提示要求遇到“今天/当前/现在/最新/实时”等问题时先取时间，再判断是否需要搜索。

**验收标准：**
- ✅ 返回日期、时间、星期、时区
- ✅ 时区读取 `rag.tools.time.default-zone`

##### web_search（联网搜索工具）

**功能描述：**
调用 Tavily 查询公开实时网页信息，返回标题、链接、摘要、发布时间等结构化结果。

**执行逻辑：**
1. API Key 只从环境变量读取
2. 搜索参数读取 `rag.tools.web-search` 与 `search.*`
3. 搜索结果用于模型生成答案，并由后端转换为网页引用

**验收标准：**
- ✅ 搜索实时公开信息时返回结构化结果
- ✅ 搜索结果包含来源链接
- ✅ 未配置 API Key 时返回可解释失败，不编造实时数值
- ✅ 搜索结果可转换为 `sourceType=web` 的引用

### 2.4 Skill 系统（优先级：P1）

#### 2.4.1 Skill 框架

**功能描述：**
Skill 是有状态的多轮任务型对话模块，用于完成需要多步骤交互的任务（如发送邮件、查询天气）。

**状态管理：**
- 每个 Skill 维护独立的会话状态（Session 级别）
- 状态数据以 JSON 格式存储在数据库（`skill_states` 表）或 Redis
- 状态包含：当前步骤、已收集的数据、任务进度

**多轮对话流程：**
```
用户触发 Skill → Skill 询问第 1 步信息 → 用户回答 → Skill 保存状态并询问第 2 步 → ... → 任务完成 → 返回 RAG 模式
```

**触发方式：**
1. **显式触发**：用户输入关键词（如"我要发邮件"）
2. **LLM 识别**：LLM 判断用户意图并自动触发 Skill

**验收标准：**
- ✅ 支持多个 Skill 同时注册
- ✅ Skill 状态在会话期间持久化（刷新页面不丢失）
- ✅ Skill 完成后自动返回 RAG 模式
- ✅ Skill 执行超时保护（30 分钟无交互自动清除状态）
- ✅ 支持用户中途退出 Skill（输入"取消"或"退出"）

#### 2.4.2 示例 Skill

##### EmailSkill（邮件发送 Skill）

**功能描述：**
通过多轮对话收集邮件信息（收件人、主题、内容），最后确认并发送邮件。

**状态机：**
```
INIT → ASK_RECIPIENT → ASK_SUBJECT → ASK_CONTENT → CONFIRM → SEND → DONE
```

**对话流程：**
1. **INIT**：用户输入"我要发邮件"
   - 系统：`好的，请告诉我收件人的邮箱地址。`
2. **ASK_RECIPIENT**：用户输入 `user@example.com`
   - 系统：`收件人已设置为 user@example.com。请输入邮件主题。`
3. **ASK_SUBJECT**：用户输入 `会议通知`
   - 系统：`邮件主题已设置为"会议通知"。请输入邮件内容。`
4. **ASK_CONTENT**：用户输入 `明天下午 2 点开会`
   - 系统：`邮件内容已设置。请确认发送：\n收件人：user@example.com\n主题：会议通知\n内容：明天下午 2 点开会\n\n回复"确认"发送，"取消"放弃。`
5. **CONFIRM**：用户输入 `确认`
   - 系统：`邮件已发送成功！`（状态转为 DONE）

**状态数据示例：**
```json
{
  "step": "ASK_CONTENT",
  "recipient": "user@example.com",
  "subject": "会议通知",
  "content": null
}
```

**验收标准：**
- ✅ 完整走完邮件发送流程（5 个步骤）
- ✅ 每个步骤正确验证用户输入（邮箱格式、非空检查）
- ✅ 支持中途取消（清除状态）
- ✅ 邮件发送成功后返回 RAG 模式

##### WeatherSkill（天气查询 Skill）

**功能描述：**
通过多轮对话收集城市和日期信息，调用天气 API 返回天气预报。

**状态机：**
```
INIT → ASK_CITY → ASK_DATE → QUERY_API → DONE
```

**对话流程：**
1. **INIT**：用户输入"查询天气"
   - 系统：`请告诉我你想查询哪个城市的天气？`
2. **ASK_CITY**：用户输入 `北京`
   - 系统：`请告诉我你想查询哪一天的天气？（今天/明天/后天）`
3. **ASK_DATE**：用户输入 `明天`
   - 系统调用天气 API，返回：`北京明天天气：晴，气温 15-25℃，风力 3 级。`

**验收标准：**
- ✅ 完整走完天气查询流程（3 个步骤）
- ✅ 支持常见城市名称（中文）
- ✅ 支持相对日期（今天/明天/后天）
- ✅ 天气 API 调用失败时返回友好提示

### 2.5 用户界面（优先级：P0）

#### 2.5.1 登录/注册界面

**功能描述：**
提供登录和注册入口，未登录用户访问系统时自动跳转到登录页面。

**页面布局：**
- 登录页面：用户名/邮箱、密码输入框、"记住我"复选框、登录按钮、"还没有账号？立即注册"链接
- 注册页面：用户名、邮箱、密码、确认密码输入框、注册按钮、"已有账号？立即登录"链接

**交互流程：**
1. 用户访问系统 → 未登录 → 自动跳转到登录页面
2. 点击"立即注册" → 进入注册页面 → 填写信息 → 注册成功 → 自动登录 → 跳转到主界面
3. 点击"立即登录" → 填写信息 → 登录成功 → 跳转到主界面

**验收标准：**
- ✅ 表单实时验证（用户名格式、邮箱格式、密码强度）
- ✅ 登录/注册失败时显示明确的错误提示
- ✅ 响应式布局（适配桌面端）
- ✅ 密码输入框支持显示/隐藏密码

#### 2.5.2 文档管理界面

**功能描述：**
登录用户可以上传、查看、删除自己的文档。

**页面布局：**
- 顶部：用户信息区域（头像、用户名、退出登录按钮）+ 上传按钮 + 搜索框
- 中部：文档列表（表格）
  - 列：文件名、类型、大小、上传时间、状态、操作（查看详情、删除）
- 底部：分页控件

**交互流程：**
1. 点击"上传文档"按钮 → 选择文件 → 上传进度条 → 上传成功提示
2. 点击"删除"按钮 → 确认对话框 → 删除成功提示

**验收标准：**
- ✅ 支持拖拽上传文件
- ✅ 上传进度条实时更新
- ✅ 文档列表支持分页（每页 20 条）
- ✅ 删除操作需要二次确认
- ✅ 响应式布局（适配桌面端）

#### 2.5.3 对话界面

**功能描述：**
登录用户可以在对话框中输入问题，系统基于用户自己的知识库返回答案。

**页面布局：**
- 左侧：会话列表（可切换历史会话）
- 右侧：对话区域
  - 顶部：会话标题 + 新建会话按钮
  - 中部：消息列表（用户消息 + 系统回答）
  - 底部：输入框 + 发送按钮

**消息格式：**
- 用户消息：右对齐，蓝色背景
- 系统回答：左对齐，灰色背景
- 回答包含引用来源（文档名称，点击可跳转）

**交互流程：**
1. 输入问题 → 点击发送 → 显示"正在思考..."加载动画 → 返回答案
2. 答案支持 Markdown 渲染（代码块、列表、加粗等）

**验收标准：**
- ✅ 支持回车键发送消息
- ✅ 消息列表自动滚动到底部
- ✅ 支持复制消息内容
- ✅ 支持 Markdown 渲染
- ✅ 引用来源可点击（显示文档详情）

#### 2.5.4 工具/Skill 管理界面（管理员）

**功能描述：**
管理员可以启用/禁用工具。工具运行参数由后端配置文件和环境变量维护，前端不编辑 JSON 参数。

**页面布局：**
- 工具列表（卡片形式）
  - 每个卡片包含：工具名称、描述、是否可用、启用/禁用开关
- 提供刷新按钮重新加载当前状态

**验收标准：**
- ✅ 工具启用/禁用实时生效（无需刷新页面）
- ✅ 普通用户不可访问工具管理页
- ✅ 不暴露 JSON 参数、Hook 或优先级编辑

## 3. 非功能需求

### 3.1 性能需求

- **文档上传处理速度：** 10MB 文档在 30 秒内完成分块和向量化
- **检索响应时间：** 用户提问后 3 秒内返回答案（正常网络条件）
- **并发支持：** 支持 50 个并发用户同时对话
- **向量检索速度：** Top-K 检索在 100ms 内完成（知识库 < 10 万条向量）

### 3.2 安全需求

- **JWT 认证：** Token 有效期 2 小时，勾选"记住我"后 7 天
- **密码加密：** 使用 BCrypt 算法加密存储
- **API Key 安全存储：** LLM API Key 加密存储，不提交到 Git
- **权限控制：** ADMIN 角色可配置工具，USER 角色仅可使用
- **输入验证：** 防止 SQL 注入、XSS 攻击
- **文件上传限制：** 单个文件最大 50MB，禁止上传可执行文件
- **数据隔离：** 用户只能访问自己的文档和对话历史

### 3.3 可扩展性

- **工具动态加载：** 新增工具无需修改核心代码
- **Skill 扩展机制：** 新增 Skill 仅需实现 `Skill` 接口
- **向量库性能：** Milvus 支持海量向量数据和高性能检索
- **LLM API 可切换：** 支持多种 LLM API（DeepSeek / 智谱 / OpenAI）

### 3.4 可维护性

- **代码规范：** 遵循 CLAUDE.md 定义的分层架构和命名规范
- **日志记录：** 关键操作记录日志（文档上传、检索、工具执行、错误）
- **错误处理：** 统一异常处理机制（`@RestControllerAdvice`）
- **单元测试：** 核心模块测试覆盖率 > 70%

## 7. 术语表

| 术语 | 英文 | 解释 |
|------|------|------|
| RAG | Retrieval-Augmented Generation | 检索增强生成，结合向量检索和 LLM 生成答案的技术 |
| Embedding | Embedding | 向量嵌入，将文本转换为高维向量的过程 |
| Chunk | Chunk | 文档分块，将长文本切分为固定大小的片段 |
| LLM | Large Language Model | 大语言模型，如 GPT-4、DeepSeek、智谱 GLM 等 |
| Tool | Tool | 工具，可在 RAG 前后执行自定义逻辑的独立模块 |
| Skill | Skill | 技能，有状态的多轮任务型对话模块 |
| Top-K | Top-K | 检索结果中相似度最高的 K 个结果 |
| Prompt | Prompt | 提示词，发送给 LLM 的输入文本 |
